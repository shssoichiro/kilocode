# JetBrains Session Test Plan

## Goal

Change the JetBrains frontend session tests from many small event-only assertions toward readable string snapshots built into the production types themselves, with a transcript-first `SessionController.toString()` and a compact status line that reflects readiness and runtime state. Permission and question data should be rendered in the same order the real UI presents them: after the message transcript, at the bottom of the visible content, just above the final status line.

## Scope

1. Rename `SessionManagerTestBase` to `SessionControllerTestBase`.
2. Remove stored `SessionModel.ready` and replace it with a computed readiness method based on app/workspace status.
3. Ensure app/workspace state changes applied to `SessionModel` happen on the EDT immediately before their controller event is fired.
4. Add readable `toString()` on `SessionModel` and `SessionController`.
5. Add assertion helpers so tests compare multiline expected strings and separately verify whether messages are visible.
6. Convert permission/question assertions to use the `toString()` output instead of field-by-field state extraction.
7. Convert or add tests to use the new string representation.

## Current State

- The existing base class is `packages/kilo-jetbrains/frontend/src/test/kotlin/ai/kilocode/client/session/model/SessionManagerTestBase.kt`.
- It is already controller-oriented: it creates `SessionController`, wires fake RPC services, exposes `prompted()`, `flush()`, and DTO factories.
- 11 frontend tests extend this base.
- `SessionModelTest` is separate and tests the pure model directly.
- There is no existing snapshot or golden assertion helper in `packages/kilo-jetbrains`.
- The nearest local convention is inline triple-quoted strings with `trimIndent()` and plain `assertEquals()`.
- `SessionModel` currently stores:
  - `app`
  - `workspace`
  - `ready`
  - `showMessages`
  - message content and runtime state
- `SessionController` currently mutates `model.app` and `model.workspace` inside `edt { ... }`, then fires `AppChanged` / `WorkspaceChanged`.
- `SessionController` currently sets `model.ready = true` when workspace status becomes `READY`.
- Permission/question tests still contain field-by-field assertions like:
  - cast `model.state` to `AwaitingPermission` or `AwaitingQuestion`
  - assert `patterns`, `always`, `tool`, `filePath`, and other fields individually

## Recommended Implementation

### 1. Rename The Base

Rename:

- File: `SessionManagerTestBase.kt` -> `SessionControllerTestBase.kt`
- Class: `SessionManagerTestBase` -> `SessionControllerTestBase`

Update all 11 subclasses to extend the new name.

Note:

- I will assume the intended final spelling is `SessionControllerTestBase` even though the request contains typos.
- I would not move packages or directories in this pass. The tests currently live under `session/model`, but the requested change only requires renaming the base class and updating usage.

### 2. Make Readiness Computed, Not Stored

Remove `SessionModel.ready`.

Add a method on `SessionModel` that computes readiness from the app and workspace state, for example:

- `fun isReady(): Boolean`

Planned logic:

- app is ready when `app.status == KiloAppStatusDto.READY`
- workspace is ready when `workspace.status == KiloWorkspaceStatusDto.READY`
- overall readiness is true only when both are ready

This removes duplicated state and avoids drift between `ready` and the two underlying statuses.

### 3. Update App/Workspace Model State On EDT Right Before Firing

For app/workspace changes, mutate the model only on the EDT, then fire the matching controller event.

Requested approach:

- extend `fire()` with an optional closure that runs immediately before listeners are notified
- when already on the EDT, run the closure first and then notify listeners
- when not on the EDT, schedule both together so the mutation and event delivery still happen on the EDT in order

Planned shape:

- `private fun fire(event: SessionControllerEvent, before: (() -> Unit)? = null)`

Use it for app/workspace updates like this conceptually:

- `fire(SessionControllerEvent.AppChanged) { model.app = state; model.version = app.version }`
- `fire(SessionControllerEvent.WorkspaceChanged) { model.workspace = state; ... }`

This gives two guarantees:

- UI reads the updated model from the EDT
- listeners observe the event only after the model was updated on that same EDT turn

### 4. Add Readable `toString()` Implementations

The formatter should live in production types, not only in test helpers.

Planned shape:

- `SessionModel.toString()`
- `SessionController.toString()`

Recommended responsibilities:

- `SessionModel.toString()`:
  - render message content and current interaction state
  - include reasoning, tools, text, compaction, question, and permission data
  - place question/permission blocks after the message transcript, matching the UI order
  - omit app/workspace readiness noise
- `SessionController.toString()`:
  - render the full user-facing transcript plus a final one-line status summary
  - if the controller is not ready, summarize app/workspace status instead of repeating runtime status details

### 5. Separate Dump Structure From Visibility Assertion

Do not include visibility inside the dump string body.

Instead:

- the dump string should exclude message visibility metadata
- assertion helpers should separately verify whether messages are visible

Planned helper shape:

- `protected fun assertController(expected: String, c: SessionController, show: Boolean = true)`
- `protected fun assertModel(expected: String, m: SessionModel)`
- helper compares normalized string output
- helper then checks `c.model.showMessages == show` for controller assertions

This keeps the dump readable and avoids noisy lines like `View: Messages` in every assertion.

### 6. New String Format

The controller dump should look like a real transcript first.

Top section:

- all messages in order
- no `Messages` header
- include reasoning, tool calls, text, and compaction entries

Then, at the bottom of the transcript content:

- if a question exists in current state, include it after the messages as its own block
- if a permission request exists in current state, include it after the messages as its own block
- this is where the real UI exposes those pending interactions
- separate top-level transcript blocks with `---`

Bottom section:

- one final status line

If ready:

- `[code] [kilo/gpt-5] [busy] [running commands]`

If not ready:

- do not include message visibility or runtime status text there
- include agent/model first when already known
- then summarize app/workspace readiness, for example:
  - `[code] [kilo/gpt-5] [app: READY] [workspace: PENDING]`
  - if agent/model are still unknown: `[app: READY] [workspace: PENDING]`

### 7. Proposed Text Shape

Ready + busy example:

```text
user#msg1
text#p1:
  go
---
assistant#msg2
reasoning#p2:
  Need to inspect files first
tool#p3 bash [RUNNING] ls
text#p4:
  I found the issue

[code] [kilo/gpt-5] [busy] [running commands]
```

Question example:

```text
user#msg1
text#p1:
  go
---
question#q1
tool: msg1/call1
header: Choice
prompt: Pick one
option: A - Option A
multiple: false
custom: true

[code] [kilo/gpt-5] [awaiting-question]
```

Permission example:

```text
user#msg1
text#p1:
  go
---
permission#perm1
tool: msg1/call1
name: edit
patterns: *.kt
always: src/**
file: src/A.kt
state: RESPONDING
metadata: kind=edit

[code] [kilo/gpt-5] [awaiting-permission]
```

Not ready example:

```text
[code] [kilo/gpt-5] [app: READY] [workspace: PENDING]
```

If agent/model are not known yet:

```text
[app: READY] [workspace: PENDING]
```

Or, if there is already transcript content but readiness is still incomplete:

```text
user#msg1
text#p1:
  go

[code] [kilo/gpt-5] [app: READY] [workspace: PENDING]
```

### 8. Permission/Question Assertions Through `toString()`

Replace direct field assertions like this:

```kotlin
val state = model.state as SessionState.AwaitingPermission
assertEquals(listOf("*.kt"), state.permission.patterns)
assertEquals(listOf("src/**"), state.permission.always)
assertEquals("src/A.kt", state.permission.meta.filePath)
assertEquals(PermissionRequestState.RESPONDING, state.permission.state)
assertEquals("msg1", state.permission.tool!!.messageId)
```

with a string assertion that renders the same information in UI order, for example:

```kotlin
assertModel(
    """
    user#msg1
    text#p1:
      go
    ---
    permission#p1
    tool: msg1/call1
    name: edit
    patterns: *.kt
    always: src/**
    file: src/A.kt
    state: RESPONDING
    metadata: kind=edit
    """,
    model,
)
```

Do the same for question data.

This turns low-level state assertions into readable UI-like snapshots while still covering the same fields.

### 9. `SessionModel.toString()` Role

`SessionModel.toString()` should stay useful, but it does not need to be the main integration dump anymore.

Recommended behavior:

- same transcript content formatting used by `SessionController.toString()`
- no readiness line based on app/workspace
- no visibility information
- question/permission blocks still appear after message content
- optionally a compact final state line derived only from `SessionState` if useful for model-only tests

This keeps model tests focused on model content/state while controller tests cover readiness and visibility separately.

### 10. Formatting Rules

- Preserve message order and part order using the existing `LinkedHashMap` order.
- Separate top-level transcript blocks with `---`.
- Render multiline text with an indented block under the entry header.
- Use `<none>` for empty optional values to keep diffs explicit.
- Keep status words normalized and lowercase in brackets, for example `busy`, `idle`, `awaiting-question`.
- Include useful permission/question details directly in the dump:
  - permission: id, tool ref, name, patterns, always, file path if present, request state if present, metadata
  - question: id, tool ref, header, prompt, options, multiple/custom flags
- Avoid raw DTO dumps, autogenerated data-class output, or transport-layer formatting because they are harder to scan and less stable.

### 11. Add Snapshot-Style Tests Around Controller Behavior

Recommended first conversions or additions:

- `PromptLifecycleTest`
  - permission asked -> assert controller string shows transcript + permission block + final status line
  - question asked -> assert controller string shows transcript + question block + final status line
  - question rejected -> assert controller string returns to transcript + idle status line
- `TurnLifecycleTest`
  - turn open -> assert controller string shows busy status line
  - error -> assert controller string shows error state in the final line or error section if needed
- `StatusComputationTest`
  - tool part update with `bash` -> assert final line becomes `[code] [kilo/gpt-5] [busy] [running commands]`
- `MessageListTest`
  - assistant message with text / reasoning / tool parts -> assert transcript block formatting
- `HistoryLoadingTest`
  - loaded history appears in the transcript exactly as the UI would render it
- `AppWatchingTest` / `WorkspaceWatchingTest`
  - assert not-ready line shape such as `[app: READY] [workspace: PENDING]`
- `SessionModelTest`
  - convert permission/question unit assertions to `assertModel(...)` snapshots when the goal is to verify visible content, not raw object storage
- visibility checks
  - assertion helper verifies `showMessages` separately instead of embedding it in the dump string

The existing fine-grained assertions do not all need to disappear in one pass. The safe approach is:

- keep narrow unit tests in `SessionModelTest` where object identity or event behavior is the real subject
- add or convert scenario tests to assert the full readable dump
- keep visibility as a separate assertion
- prefer `toString()` assertions for permission/question content that is meant to mirror the UI

## Risks And Mitigations

- Risk: dump strings become too noisy.
  - Mitigation: keep transcript first, visibility separate, and status compressed into one final line.
- Risk: readiness could become inconsistent if still stored separately.
  - Mitigation: remove `ready` and derive readiness from app/workspace statuses only.
- Risk: UI could read app/workspace state off-EDT.
  - Mitigation: apply app/workspace model mutations inside the EDT fire path before notifying listeners.
- Risk: tests become brittle if wording changes.
  - Mitigation: keep labels stable, lowercase normalized state words, and compact bracket syntax.
- Risk: permission/question assertions lose coverage of important fields.
  - Mitigation: include the important fields directly in the `toString()` format so the snapshot still covers them.

## Recommended Final Shape

1. Rename the base to `SessionControllerTestBase`.
2. Remove stored `SessionModel.ready` and add computed `isReady()`.
3. Update `SessionController.fire()` to support a `before` closure executed on the EDT before listener notification.
4. Move app/workspace model mutation into that EDT fire path.
5. Add transcript-style `toString()` implementations.
6. Place permission/question blocks after the messages, matching the real UI order.
7. Replace field-by-field permission/question assertions with `toString()` snapshot assertions.
8. Keep visibility checks out of the dump and assert them separately.
9. Convert controller tests to use the new transcript + one-line-status format.
