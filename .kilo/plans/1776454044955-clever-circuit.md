# JetBrains Frontend Session State Cleanup

## Goal

Replace the current `SessionPhase` + `PromptState` + `StatusState` stack with one explicit session state model that answers a single question cleanly: `what is this session doing right now?`

## Why The Current Model Feels Messy

### Current shape

- `SessionModel.phase: SessionPhase`
- `SessionPhase.Working(val status: StatusState)`
- `SessionPhase.Prompting(val prompt: PromptState)`
- `StatusState` has only `Thinking(text)` and `Working(text)`
- `PromptState` has only `Asking(question)` and `Permitting(permission)`

### Problems

1. The real state is split across three types.
   `SessionPhase` alone is not enough; callers often need the nested payload type too.

2. `StatusState` does not carry independent behavior.
   In practice the UI only reads `text`, and the `Thinking` vs `Working` distinction is not used meaningfully.

3. `PromptState` is a wrapper around already meaningful domain models.
   `Question` and `Permission` are already specific enough to be top-level session states.

4. `SessionUi` already treats multiple variants as one concept.
   Busy is derived with `phase is SessionPhase.Working || phase is SessionPhase.Prompting`, which is a sign the state model is indirect.

5. Naming is working against readability.
   `phase` sounds like an internal lifecycle step, while the user-facing need is a single `state`.

## Recommended Target Model

Rename the concept from `phase` to `state` and flatten it into one sealed class.

```kotlin
sealed class SessionState {
    data object Idle : SessionState()
    data class Busy(val text: String) : SessionState()
    data class AwaitingQuestion(val question: Question) : SessionState()
    data class AwaitingPermission(val permission: Permission) : SessionState()
    data class Retry(val message: String, val attempt: Int, val next: Long) : SessionState()
    data class Offline(val message: String, val requestId: String) : SessionState()
    data class Error(val message: String, val kind: String? = null) : SessionState()
}
```

### Design notes

- `Busy(text)` replaces `SessionPhase.Working(StatusState.*)`.
- `AwaitingQuestion` replaces `SessionPhase.Prompting(PromptState.Asking(...))`.
- `AwaitingPermission` replaces `SessionPhase.Prompting(PromptState.Permitting(...))`.
- `Retry`, `Offline`, and `Error` stay top-level because they represent distinct user-visible conditions.
- `StatusState` and `PromptState` are removed entirely.

## Intended Rules

At any point, `SessionModel.state` must be exactly one of:

1. `Idle`
2. `Busy`
3. `AwaitingQuestion`
4. `AwaitingPermission`
5. `Retry`
6. `Offline`
7. `Error`

No nested state machine for normal rendering. The UI should only switch on `state`.

## Transition Mapping

Map chat events directly to the new state:

| Event                           | New state                                                       |
| ------------------------------- | --------------------------------------------------------------- |
| `TurnOpen`                      | `Busy("Considering...")`                                        |
| `PartUpdated(reasoning)`        | `Busy("Thinking...")`                                           |
| `PartUpdated(text)`             | `Busy("Writing...")`                                            |
| `PartUpdated(tool=...)`         | `Busy(tool-specific text)`                                      |
| `TurnClose`                     | `Idle`                                                          |
| `Error`                         | `Error(message, kind)`                                          |
| `PermissionAsked`               | `AwaitingPermission(permission)`                                |
| `PermissionReplied`             | `Busy("Considering...")`                                        |
| `QuestionAsked`                 | `AwaitingQuestion(question)`                                    |
| `QuestionReplied`               | `Busy("Considering...")`                                        |
| `QuestionRejected`              | `Idle`                                                          |
| `SessionStatusChanged(idle)`    | `Idle`                                                          |
| `SessionStatusChanged(busy)`    | `Busy("Considering...")` only when recovering from coarse state |
| `SessionStatusChanged(retry)`   | `Retry(...)`                                                    |
| `SessionStatusChanged(offline)` | `Offline(...)`                                                  |

## Implementation Plan

### 1. Replace the state types

Files:

- `packages/kilo-jetbrains/frontend/src/main/kotlin/ai/kilocode/client/session/model/SessionPhase.kt`
- `packages/kilo-jetbrains/frontend/src/main/kotlin/ai/kilocode/client/session/model/PromptState.kt`
- `packages/kilo-jetbrains/frontend/src/main/kotlin/ai/kilocode/client/session/model/StatusState.kt`

Changes:

- Introduce `SessionState` as the single sealed class.
- Remove `PromptState`.
- Remove `StatusState`.
- Either rename `SessionPhase.kt` to `SessionState.kt` or replace its contents with the new model and then rename references.

### 2. Rename the model field and event

Files:

- `SessionModel.kt`
- `SessionModelEvent.kt`

Changes:

- Rename `phase` to `state`.
- Rename `setPhase()` to `setState()`.
- Rename `PhaseChanged` to `StateChanged`.
- Reset `state` to `Idle` in `loadHistory()` and `clear()`.

Result:

- `SessionModel` exposes exactly one current state field.

### 3. Flatten session manager transitions

File:

- `SessionManager.kt`

Changes:

- Replace every `chat.setPhase(...)` call with `chat.setState(...)`.
- Remove `toPermissionPrompt()` and `toQuestionPrompt()` returning wrapper types.
  Instead return `Permission` and `Question` directly or inline the construction.
- Replace `status(): StatusState` with `status(): String`.
- Convert all event branches to produce a direct `SessionState` variant.

Important simplification:

- There should be no code path that first decides `state = Prompting` and then asks the UI to inspect another type.
- There should be no code path that wraps a text label inside a second state class.

### 4. Update UI consumers to switch on one type

Files:

- `SessionUi.kt`
- `MessageListUi.kt`
- any prompt or status panels that react to `PhaseChanged`

Changes:

- Switch listeners from `PhaseChanged` to `StateChanged`.
- Derive busy directly from `state`:
  `Busy`, `AwaitingQuestion`, `AwaitingPermission`, `Retry`, and `Offline` are busy-like.
- Render message/footer text from `SessionState.Busy.text`.
- Render errors from `SessionState.Error.message`.

Recommendation:

- Add a small helper near the UI if needed:
  `private fun SessionState.isBusy(): Boolean`
  so the busy rule is centralized and readable.

### 5. Update tests to assert direct states

Files:

- `SessionModelTest.kt`
- `TurnLifecycleTest.kt`
- `StatusComputationTest.kt`
- `ListenerLifecycleTest.kt`
- any prompt-related tests

Changes:

- Rename assertions from `PhaseChanged` to `StateChanged`.
- Replace `SessionPhase.Working(StatusState.Working(...))` style expectations with `SessionState.Busy(...)`.
- Replace `SessionPhase.Prompting(PromptState...)` expectations with `SessionState.AwaitingQuestion(...)` or `SessionState.AwaitingPermission(...)`.

Add or update coverage for:

- `TurnOpen -> Busy`
- `TurnClose -> Idle`
- `Error -> Error`
- `PermissionAsked -> AwaitingPermission`
- `PermissionReplied -> Busy`
- `QuestionAsked -> AwaitingQuestion`
- `QuestionRejected -> Idle`
- tool-specific `PartUpdated -> Busy(text)`
- `SessionStatusChanged(retry/offline)` producing explicit states

### 6. Verify dead branches and naming fallout

Search for and remove any remaining references to:

- `SessionPhase`
- `PromptState`
- `StatusState`
- `phase`
- `PhaseChanged`

Then verify there is only one current-state concept left in the frontend session model.

## Recommended Implementation Order

1. Add `SessionState` and update `SessionModel`/`SessionModelEvent`.
2. Update `SessionManager` transitions and status helper.
3. Update UI listeners and rendering.
4. Update tests.
5. Run frontend tests.

## Expected Outcome

After the refactor, the session model should read like this:

- `chat.state` tells the full story.
- UI switches on one sealed class.
- No extra prompt/status wrapper types are needed.
- State transitions become straightforward to trace from chat events.
